# Reactive and Exploratory

## Dependencies

[Community alignment](/deliverables/alignment.md)

## Description 

We acknowledge that, given DevX's vast surface area, opportunities may emerge during the implementation of this strategy that turn out to be low-hanging fruit or high ROI (low effort with high impact on DevX). We also have several ideas to improve DevX that we're not sure are worth the effort, either because we don't know if they'd be actually useful or because we're unsure how hard they'd be to implement.

To avoid losing these opportunities, we added this output, which serves as a catch-all for every reactive and exploratory measure we identify during implementation of this strategy. 

If we encounter those opportunities, we'll have someone to execute on them. If we don't, we'll use those resources to explore potential improvements on DevX that we've already identified but aren't sure are worth it. Examples of these are:
- Exploring defining a canonical representation for CBOR.
- Creating an embedded and easy-to-use CBOR analyzer/interpreter (VSCode Extension/TUI/CLI).
- Adding an Error translation layer to the node improves node error messages for developers.
- Integrating a no-code tool to "draw" transactions and convert them to Tx3/MeshJS/other. 
- Generating off-chain types from CIP-57
- Extending CIP-57 (blueprints) to contain more data about the contracts.
- Facilitate using an embedded UPLC interpreter for error-driven off-chain development.
- Write new CIPs to improve DevX.

In the best-case scenario, we create many of these tools. In the worst-case scenario, we have compelling reasons why they weren't pursued, saving ecosystem developers a ton of work.

## Outcome 

The outcome is that we'll be able to quickly respond to developers' demands and explore (and subsequently implement or discard) many ideas we have to improve DevX.

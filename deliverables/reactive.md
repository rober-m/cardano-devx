# Reactive and Exploratory

## Dependencies

[Community alignment](/deliverables/alignment.md)

## Description 

We realize that DevX being such a vast surface area, there might be opportunities that surface during the implementation of this strategy that turn out to be low-hanging fruit or high ROI (low effort with high impact on DevX). We also have several ideas to improve DevX that we're not sure if they are worth the effort. Being because we don't know if they'd be actually useful or because we're unsure how hard they'd be to implement.

To avoid loosing these opportunities, we added this output that serves as a catch all for every reactive and exploratory measures we identify during the implementation of this strategy. 

On the case we encounter said opportunities, we'll have someone to execute on them. In the case we don't, we'll use those resources to explore potential improvements on DevX that we already identified but are unsure if they're worth it. Examples of these are:
- Exploring defining a canonical representation for CBOR.
- Creating an embedded and easy to use CBOR analyzer/interpreter (VSCode Extension/TUI/CLI).
- Adding an Error translation layer to the node improve node error messages for developers.
- Integrating a no-code tool to "draw" transactions and convert them to Tx3/MeshJS/other. 
- Generating off-chain types from CIP-57
- Extending CIP-57 (blueprints) to contain more data about the contracts.
- Facilitate using an embedded UPLC interpreter for error-driven off-chain development.
- Write new CIPs to improve DevX.

## Outcome 

The outcome is that we'll be able to quickly react to the demands of developers and explore (and subsequently implement or discard) many ideas we have to improve DevX.
